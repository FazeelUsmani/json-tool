# Benchmark methodology

How the numbers in the launch narrative were measured. Companion to [benchmarks/README.md](./README.md), which documents the fixture shapes themselves.

If a claim on the marketing surface ("200MB JSON in 6 seconds") doesn't show up in the results table below with a documented setup, the claim is unsupported. The point of this page is to make every public number reproducible by an HN commenter in 10 minutes.

Last measured: **2026-05-22** (W4-Mon). Numbers re-verify on every release.

---

## Setup

| Item               | Value                                       |
|--------------------|---------------------------------------------|
| Hardware           | Apple Silicon (M-series) MacBook            |
| OS                 | macOS 14+ (Darwin 25.4.0)                   |
| Browser            | Chrome 147 (stable channel)                 |
| Node               | 20.x (matches `.nvmrc`)                     |
| Server             | `npm run preview` (Vite SSG build; default port 4173, picks next free port if in use) |
| DevTools           | Open before drop, Console + Performance Monitor visible |

**Why `npm run preview` and not `npm run dev`:** dev mode bundles unminified React, enables React strict-mode double-effects, and serves via Vite's transform pipeline. We measured a consistent ~6.5× harness overhead between dev (or vitest harness) and the actual production bundle (see `benchmarks/parser-direct.mjs` for the math). All public numbers are from the production build.

**Why DevTools open:** Performance Monitor surfaces the JS heap line during parse, which is the only way to capture peak memory cleanly. Console reads the worker host's `[parser] parseFile <MB> → <ms>` log. No microbenchmark instrumentation is added — we measure what the user sees.

---

## Cold-run protocol

1. Quit Chrome fully (cmd-Q, not just close window). Renderer process state ("primed-to-kill" after a previous OOM) doesn't carry over.
2. Relaunch, open `http://localhost:4174/`.
3. DevTools → Application → Service Workers → check **"Update on reload"**. Hard-reload picks up the latest bundle without manual unregister. (The PWA's `skipWaiting + clientsClaim` config means this only matters for the dev loop.)
4. DevTools → Performance Monitor → enable "JS heap size" line.
5. Drag-drop the fixture file from Finder onto the tab. **Do not paste** — the paste path bypasses viewer-only mode (intentional; see Limitations). Drag-drop is the measured path.
6. Wait for the worker host log: `[parser] parseFile <MB> → <ms> (<MB/s>)`.
7. Record: parseMs from the log; heap peak from Performance Monitor; tab survival (binary).

For search benchmarks, after the parse log:
- Type the needle in the search box.
- Time from keystroke to "spinner disappears + final N/M counter stable."

## Results — 2026-05-22 (W3-Thu→Fri overnight cold-runs)

Captured against the production bundle through commit `5956906` (Memory HUD shipped, hydration regression fixed). The `?debug=1` Memory HUD provided heap + spine-count + line-count readouts directly, replacing the Performance-Monitor-only approach used on 2026-05-21. Closes two of the three "not measured" gaps below: 505 MB browser cold-run + 200 MB NDJSON browser cold-run.

### Parse + render — browser cold-run

| Fixture                       | Size   | Browser parse        | Heap peak | Heap steady | Expansion | Tab survived |
|-------------------------------|--------|----------------------|-----------|-------------|-----------|--------------|
| `telemetry-2250000.json`      | 505 MB | 13858 ms (36.4 MB/s) | 1.69 GB   | 1.16 GB     | 3.35×     | ✓            |
| `telemetry-900000.ndjson`     | 201 MB | — *                  | —         | 262.4 MB    | 1.30×     | ✓            |

\* HUD `Parse: —` bug discovered during this measurement — the NDJSON parse path runs main-thread via `parseNdjson()` and didn't go through `parserHost.parseFile`'s telemetry capture, so wall-clock time wasn't recorded by the HUD. Console `[parser] parseNdjson <MB> → <ms>` log line was also absent. **Fix lands in the same commit pass as this methodology append** — re-measure NDJSON parse time in the next session.

**Linear browser throughput confirmed at 505 MB.** 36.4 MB/s on the 505 MB fixture extrapolates cleanly from the 35.6 MB/s observed at 201 MB (2026-05-21). The streaming parser holds linear scaling across 2.5× file sizes — the constant factor is React + Comlink + structured-clone overhead, not parser nonlinearity. The earlier methodology projection of "~14 s at 505 MB" lands within 1% of the measured 13.858 s.

**Headline NDJSON insight: 2.5× lower memory pressure than equivalent JSON.** The 200 MB NDJSON heap settles at 262 MB (1.30× source); the 505 MB JSON heap settles at 1.16 GB (3.35× source). NDJSON's line-stub representation — line bytes stay in the Blob, only byte-range offsets materialize as tree nodes — costs roughly one-third the per-byte memory of fully-materialized JSON spine + stubs. Backs the launch claim "NDJSON with less memory pressure than the equivalent JSON".

The 505 MB browser run required a temporary `MAX_FILE_BYTES` bump from 500 → 520 MB in `MonacoPane.tsx` to bypass the public-claim ceiling for measurement purposes (working-tree only, never committed). Production cap is unchanged at 500 MB.

### Schema inference verification (W4-Mon close-out)

All three emit formats browser-verified end-to-end:
- **`telemetry-100.json`** (22 KB, well-formed) — JSON Schema + TypeScript + Zod render with correct required/optional/nullable mapping. Copy button respects the active sub-tab.
- **`pathological-1000.json`** (1.2 MB, fixture explicitly built to break things) — worker survives the inference; all three emitters produce syntactically valid output. Edge-case keys confirmed:
  - Mega-long word-only identifier (200+ chars) emitted bare via the safe-identifier check.
  - Empty-string key emitted as `""?: T` / `"": z.X()` — non-identifier branch quotes correctly.
- Strict-thresholding rule visible in the output: every field on the pathological fixture is `optional` because pathological records have fields absent in some samples (matches the footer chip's "Required = present in all samples · Nullable = null in any sample" disclosure).

**Known cost: TreeNode structured-clone across the schema-worker boundary.** The original commit-message claim that "Blob handle passes through Comlink trivially" is correct for the Blob argument but doesn't address the `root: TreeNode` argument — that IS structured-cloned. Extrapolating from the 2026-05-22 Pivot 2 `setFlat` checkpoint (370 ms at 38 MB, 2,530 ms at 200 MB, ~6.3 s extrapolated at 505 MB), every Schema-tab Refresh click pays this cost on the main thread. M1 acceptable since Refresh is explicit and infrequent; M2 fix routes inference through the parser worker (which already holds the parsed tree, eliminating the clone). Tracked as a follow-up slice.

### Interaction-to-Next-Paint at scale — identified and fixed

**Pre-fix observation.** During the 505 MB session with a search active, Chrome Performance Insights recorded **INP 2,000 ms** on search-input keystrokes (worst case; subsequent keystrokes 1,160–1,656 ms). The search worker scan runs off-thread per the W3-Thu slow-path design, but the synchronous `findMatches` walk over 2.25 M FlatRows + React reconciliation on each keystroke was main-thread. Same problem at 200 MB extrapolated to ~800 ms.

**Pivot considered, then abandoned.** First instinct: move `findMatches` into the parser worker so the walk happens off-thread. A measurement checkpoint killed that approach — structured-cloning the FlatRow array to the worker via `postMessage` took **2,530 ms** at 200 MB (linear extrapolation: ~6,300 ms at 505 MB), and the clone serialization runs **on the main thread itself**. Net: we'd add 2.5 s of post-parse blocking to avoid a 2 s search-time blocking. Negative ROI.

**Fix shipped.** Pre-lowercased the needle once per query and replaced per-row `String.toLowerCase().includes()` with `asciiCaseInsensitiveIncludes(haystack, needleLower)` — a tight loop that case-folds A-Z → a-z via the `c | 32` bit trick during comparison, without allocating a lowercase copy of every haystack. The prior approach allocated 2 × `flat.length` lowercase strings per keystroke (4.5 M allocations at 505 MB); the new path allocates zero per-row. ASCII-only fold matches the existing `searchStubs` worker behavior, so result sets stay consistent across the sync FlatRow walk and the worker byte scan.

| Fixture                       | Pre-fix INP worst | Post-fix INP worst | Post-fix INP typical | Improvement |
|-------------------------------|-------------------|--------------------|----------------------|-------------|
| `telemetry-900000.json` (200 MB) | ~800 ms (proj.) | **168 ms** ✓ (under "good" 200 ms threshold) | 24–96 ms | ~5× |
| `telemetry-2250000.json` (505 MB) | 2,000 ms        | **560 ms** (borderline poor, was poor) | 344–472 ms | ~4× |

**200 MB now clears Chrome's "good" INP threshold of 200 ms** on worst-case keystrokes — that's the headline B-narrative size and the strongest claim we can make. 505 MB lands in "needs improvement / borderline poor" territory; type-to-filter works but feedback is felt as ~0.5 s at the high end. Documented honestly in the B-narrative launch copy.

The reported overall INP at 505 MB (800 ms) is a **pointer** interaction on a tree row — distinct from search-keystroke INP. Tree-row click latency at 2.25 M rows is a separate cost (stub expand round-trip + react-window scroll-to-row) and out of scope for this fix.

## Results — 2026-05-21

### Parse + render

Two columns: **JS-direct** (Node 20 running the parser code with no React / Comlink / postMessage, captured via `benchmarks/parser-direct.mjs`) and **browser** (the production-bundle cold-run protocol). The browser column is what users see; the JS-direct column is the parser's intrinsic throughput, useful for isolating where time goes.

| Fixture                       | Size   | JS-direct       | Browser          | Heap peak (browser) | Tab survived |
|-------------------------------|--------|-----------------|------------------|---------------------|--------------|
| `telemetry-170000.json`       | 38 MB  | 589 ms (64 MB/s)  | ~600 ms (~63 MB/s) | <250 MB             | ✓            |
| `telemetry-900000.json`       | 201 MB | 3225 ms (62 MB/s) | 5645 ms (36 MB/s)  | <1.2 GB             | ✓            |
| `telemetry-2250000.json`      | 505 MB | 8154 ms (62 MB/s) | *projected ~14 s* | *projected ~3 GB*   | *not run*    |

**JS-direct is linear in file size at ~62 MB/s** across 38 → 505 MB, three orders of magnitude apart. Confirms the streaming parser doesn't hit any non-linear cost as files grow.

The browser column shows a consistent ~1.7–1.8× slowdown vs JS-direct, attributable to worker postMessage of the parsed tree (~225 MB structured-clone on the 201 MB fixture) plus React's render of the FlatRow array. We accepted this as the cost of the viewer; future optimization (typed-array transfer, worker-side flatten) is tracked but out of scope for the launch claim.

The 505 MB browser row is a *projection*: 1.75× of JS-direct ≈ 14 seconds. We need a real browser cold-run on the 505 MB fixture (file: `benchmarks/corpus/telemetry-2250000.json`, generated 2026-05-21) before quoting a 500 MB number on any marketing surface. **PLAN.MD's "500 MB JSON loads in <60 s" criterion is supported by the architecture but not yet directly measured.**

### Search

After parse completes, type a single needle in the search box. Worker decodes the buffer once, byte-level case-insensitive scan over each stub byte range, streaming matches back in 2000-row batches.

| Fixture                  | Stub ranges | Needle  | Matches | Scan time      |
|--------------------------|-------------|---------|---------|----------------|
| `telemetry-900000.json`  | 900 000     | `error` | ~160 K  | subjectively fast (target ~2–3 s) |

The previous implementation (per-range `TextDecoder.decode().toLowerCase().includes()`, 900K string allocations) measured at 5–10 s on the same fixture. The byte-level rewrite (`c3a0747`) replaces it with one in-place ASCII lowercase pass + naive byte-comparison per range, projected to land in the 1.5–2.5 s range. We don't have a captured wall-clock timing for the new path yet — that's a measurement gap to close before quoting the number on any marketing surface.

## Known limitations affecting numbers

1. **Paste path bypasses viewer-only mode.** Pasting JSON into Monaco crashes the tab past ~100 MB because Monaco tokenizes the full string on the main thread. This is intentional — Monaco's defaults are tuned for editing code, not viewing pathologically large JSON, and intercepting paste before Monaco sees it would fight Monaco's own handlers. Drag-drop is the measured path and the one we recommend.
2. **Search slow path is content-scan, not index-based.** Worker scans 200 MB linearly. Acceptable at 200 MB (2–3 s); will become noticeable at 500 MB+ (5–8 s projected). Index-based search (build inverted index during parse) is a future M2 optimization.
3. **ASCII-only case-folding in search.** Byte-level lowercase only handles `A-Z → a-z`. Matches JavaScript's default `String.prototype.toLowerCase()` behavior — Turkish `İ`, German `ß`, and similar locale-specific folds are preserved either way. Documented in worker.
4. **No per-tab memory cap enforced.** We rely on viewer-only mode + the parser's depth-2 spine + stub sampling to keep the working set bounded. Chrome's per-tab ceiling on 8 GB Macs is around 1.5–2 GB; 200 MB lands well under that. 500 MB is the projected ceiling on 8 GB hardware.
5. **URL load cap is 100 MB, not 500 MB.** Inconsistent with the file-drop hero claim. Tracked as a fetch-side fix item.

## Reproducing these numbers

```bash
# 1. Clone + install
git clone https://github.com/fazeelai/json-tool
cd json-tool
nvm use                    # picks up .nvmrc
npm ci

# 2. Generate fixtures
npm run bench:gen:all-dev  # ~50 MB versions of every shape
node benchmarks/generate.mjs --shape telemetry --size 900000  # 200 MB headline

# 3. Production build + serve
npm run build
npm run preview            # http://localhost:4174

# 4. In Chrome: cold-run protocol above
# 5. Compare your parseMs to the table; report mismatches in the issue tracker
```

For non-browser parser-only timing (no React, no Comlink — useful for isolating where time goes):

```bash
node benchmarks/parser-direct.mjs   # JS-direct measurement, prints MB/s
```
